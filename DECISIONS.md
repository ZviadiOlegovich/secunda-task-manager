# Архитектурные решения

Документ фиксирует решения по неоднозначным местам в REQUIREMENTS.md.

---

## 1. Аутентификация

**TTL JWT-токена** выносится в переменную окружения (`JWT_TTL`).

**Refresh-токен** реализуется (`POST /api/v1/refresh`).

**Logout** не реализуется — при необходимости пользователь просто логинится заново.

---

## 2. Видимость задач

Задачи изолированы по командам. Пользователь видит задачи только тех команд, в которых состоит. `team_id` обязателен при создании задачи. Пользователь без команды не может создавать задачи.

---

## 3. Ролевая модель

Роли: `owner`, `admin`, `member`.

| Действие | owner | admin | member |
|---|:-:|:-:|:-:|
| Создать команду | ✓ | — | — |
| Пригласить в команду | ✓ | ✓ | — |
| Создать задачу | ✓ | ✓ | ✓ |
| Обновить задачу | ✓ | ✓ | только если `created_by` |
| Просмотр задач команды | ✓ | ✓ | ✓ |

`member` может обновить задачу только если он её создатель (`created_by`).

---

## 4. Поля задачи

**Обязательные:** `title`, `team_id`, `created_by`, `status`.

**Опциональные:** `description`, `assignee_id`, `priority`, `due_date`.

**Статусы** (типизированные константы в коде): `todo`, `in_progress`, `done`.

**Приоритеты** (типизированные константы в коде): `low`, `medium`, `high`.

---

## 5. API эндпоинты задач

`DELETE /api/v1/tasks/{id}` — **не реализуется**.

`GET /api/v1/tasks/{id}` — **не реализуется**. Необходимые поля задачи возвращаются в списке. История изменений доступна отдельным запросом `GET /api/v1/tasks/{id}/history`.

---

## 6. История изменений (task_history)

Дельта-подход — одна строка на каждое изменённое поле (как в Jira).

**Структура записи:**
| Поле | Тип | Описание |
|---|---|---|
| `field_name` | VARCHAR | Имя поля: "status", "title", "assignee_id", ... |
| `old_value` | TEXT | Предыдущее значение |
| `new_value` | TEXT | Новое значение |
| `changed_by` | FK → users.id | Кто изменил |
| `changed_at` | TIMESTAMP | Когда изменил |

При изменении N полей за один `PUT` — вставляется N строк.

---

## 7. Приглашения в команду

**Упрощение:** `POST /api/v1/teams/{id}/invite` сразу добавляет пользователя в команду. Email-сервис вызывается как уведомление (fire-and-forget), circuit breaker защищает этот вызов.

**Правильная реализация на проде** (описана в README):
- Invitation-сервис публикует событие `user.invited` в очередь (Kafka / RabbitMQ)
- Email-воркер читает из топика и отправляет письма
- Ошибочные сообщения уходят в DLQ (Dead Letter Queue) — отдельный топик или таблица `notification_errors`
- Circuit breaker защищает вызов SMTP-провайдера внутри воркера

---

## 8. Комментарии к задачам

Реализуются только два эндпоинта (любой член команды):
- `POST /api/v1/tasks/{id}/comments` — добавить комментарий
- `GET /api/v1/tasks/{id}/comments` — получить список

Удаление и редактирование не реализуются.

---

## 9. Redis-кеширование

Версионированный кеш + mutex lock при промахе кеша.

```
Ключ кеша:   tasks:team:{team_id}:v:{version}:{hash(filters+page+limit)}
Версия:      team:{team_id}:cache_version  →  INCR при любом изменении задачи
TTL:         5 минут
Mutex lock:  SET lock:team:{team_id} NX EX 5
```

**Как работает invalidation:** при `POST`/`PUT` задачи делается `INCR team:{team_id}:cache_version`. Старые ключи с предыдущей версией естественно истекают по TTL — без KEYS/SCAN/DEL по паттерну.

**Mutex lock:** при промахе кеша первый запрос захватывает lock, идёт в БД и пишет кеш. Остальные параллельные запросы ждут и затем читают из кеша. Защищает от cache stampede.

**Описать в README как возможные улучшения:** probabilistic early expiration, stale-while-revalidate.

## 10. Сложные SQL-запросы — эндпоинты

Три отдельных эндпоинта, доступ только для `owner`/`admin` команды:

| Эндпоинт | Описание |
|---|---|
| `GET /api/v1/stats/teams` | Название команды, кол-во участников, кол-во done-задач за 7 дней |
| `GET /api/v1/stats/top-users` | Топ-3 пользователей по кол-ву созданных задач в каждой команде за месяц |
| `GET /api/v1/admin/integrity` | Задачи, где assignee не является членом команды |

## 11. Email-сервис и Circuit Breaker

Внутренний интерфейс `EmailService` с mock-реализацией:

```go
type EmailService interface {
    SendInvite(ctx context.Context, email string, teamName string) error
}
```

Circuit breaker (библиотека `gobreaker`) оборачивает вызов `SendInvite`. При N ошибках подряд breaker открывается — вызовы не проходят до восстановления.

## 12. Тестирование

**Unit-тесты:** сервисный слой, happy path + ключевые edge cases. Репозитории мокируются через интерфейсы. Целевое покрытие: 85% методов сервисного слоя.

**Интеграционные тесты:** repository-слой через testcontainers (изолированный MySQL-контейнер). Запускаются через build tag:
```
go test -tags integration ./...
```

Тесты никогда не трогают prod-БД. В CI/CD — отдельный шаг.

---

## 13. Пагинация

LIMIT/OFFSET везде: задачи, комментарии, история изменений.

**Параметры запроса:** `?page=1&limit=20`

**Описать в README как улучшение:** cursor-based пагинация по `id` — не имеет проблем со "съезжанием" страниц при вставке новых записей и работает за O(log n) вместо O(n) при больших offset.

---

## 14. Rate Limiting

Комбинированный подход, счётчики в Redis, лимит 100 запросов/мин:

| Эндпоинт | Ключ в Redis |
|---|---|
| `/register`, `/login` | `rate:ip:{ip_address}` |
| Все авторизованные | `rate:user:{user_id}` |

TTL 60 сек, INCR на каждый запрос. При превышении — `429 Too Many Requests`.

**Почему комбинированный:** на `/login` и `/register` user_id ещё нет — нужна защита от брутфорса по IP. Для авторизованных эндпоинтов user_id точнее: не страдают пользователи за одним корпоративным IP, нельзя обойти сменой IP.

**Описать в README как альтернативу:** in-memory (`golang.org/x/time/rate`) — проще, достаточно если сервис однoinстансный и rate limiting не критичен для безопасности.
